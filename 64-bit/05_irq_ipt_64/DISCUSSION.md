# Chapter 5: Hardware Interrupts and the System Timer

In the previous chapter, the operating system learned how to respond to processor exceptions. Whenever the CPU encountered an exceptional condition, such as a divide-by-zero operation or an invalid instruction, it consulted the Interrupt Descriptor Table and transferred control to the appropriate handler.

Those interrupts, however, originated from the processor itself.

An operating system must also respond to events generated by external hardware. A keyboard reports that a key has been pressed. A network card signals that a packet has arrived. A disk controller indicates that a read operation has completed. Most importantly, a timer generates periodic interrupts that allow the operating system to measure time and eventually schedule processes.

This chapter introduces the hardware interrupt mechanism used by traditional x86 systems. It explains how interrupt requests are delivered to the processor, why the Programmable Interrupt Controller must be reconfigured, and how the Programmable Interval Timer provides the kernel with its first notion of time.

By the end of the chapter, the kernel will no longer execute continuously from start to finish. Instead, it will periodically be interrupted by hardware, demonstrating that the operating system can react to events occurring independently of its own execution.

---

# Exceptions Versus Hardware Interrupts

The processor responds to two fundamentally different classes of events.

The first consists of **exceptions**, which are generated internally by the CPU. These occur when the processor detects an unusual condition while executing an instruction.

Examples include division by zero, invalid opcodes, page faults, and breakpoint instructions.

The second consists of **hardware interrupts**, which originate outside the processor. These are generated by peripheral devices requesting the processor's attention.

Although both eventually invoke entries in the Interrupt Descriptor Table, their origins are entirely different.

```text
                   Processor

        +-------------------------+
        |                         |
        |   Interrupt Descriptor   |
        |         Table            |
        +------------+------------+
                     |
      +--------------+--------------+
      |                             |
      v                             v
 CPU Exception              Hardware Interrupt
 (Internal Event)          (External Device)
```

From the processor's perspective, both are simply interrupts. From the operating system's perspective, however, they serve very different purposes.

Exceptions indicate that something unexpected occurred while executing software. Hardware interrupts notify the operating system that an external device requires service.

---

# Why Hardware Cannot Interrupt the Processor Directly

One might imagine connecting every hardware device directly to the processor.

Such a design quickly becomes impractical.

Modern computers contain many devices capable of requesting processor attention, including timers, keyboards, storage controllers, network interfaces, USB controllers, and many others. Allowing every device to communicate directly with the processor would require an enormous number of dedicated interrupt lines.

Instead, traditional x86 systems employ an intermediary known as the **Programmable Interrupt Controller (PIC)**.

The PIC collects interrupt requests from hardware devices and forwards them to the processor in an organized manner.

Conceptually, the communication path is as follows.

```text
Hardware Device
        |
        v
+----------------------+
| Programmable         |
| Interrupt Controller |
+----------------------+
        |
        v
       CPU
        |
        v
       IDT
```

The PIC acts as a traffic controller, ensuring that interrupt requests are delivered in an orderly fashion.

Although modern systems often replace the PIC with the Advanced Programmable Interrupt Controller (APIC), understanding the original PIC provides an excellent introduction to hardware interrupt handling.

---

# Interrupt Request Lines

The Programmable Interrupt Controller manages sixteen interrupt request lines, commonly abbreviated as **IRQs**.

Each hardware device is traditionally assigned one of these lines.

```text
IRQ0   System Timer
IRQ1   Keyboard
IRQ2   Cascade
IRQ3   Serial Port
IRQ4   Serial Port
...
IRQ14  Primary Disk
IRQ15  Secondary Disk
```

These assignments are historical rather than architectural, but they remain useful for educational operating systems.

Whenever a device requires attention, it activates its corresponding interrupt request line.

The PIC then determines when the processor should be interrupted.

---

# Why the PIC Must Be Remapped

A problem immediately arises.

By default, the PIC delivers interrupts using vectors that overlap with the processor's exception vectors.

```text
CPU Exceptions

0 ---------------- 31


Default PIC

0 ----------- 15
```

This creates an ambiguity.

If interrupt vector 8 occurs, is it a processor exception or a hardware interrupt?

The processor cannot distinguish between them because both occupy the same region of the Interrupt Descriptor Table.

The solution is **PIC remapping**.

Instead of using vectors 0 through 15, the operating system instructs the PIC to begin delivering interrupts at vector 32.

```text
CPU Exceptions

0 ------------------------ 31


Hardware IRQs

32 ----------------------- 47
```

The two groups no longer overlap.

Exceptions remain in the lower portion of the table, while hardware interrupts occupy their own dedicated entries.

This organization has become the standard layout used by virtually every x86 operating system.

---

# Extending the Interrupt Descriptor Table

The previous chapter introduced the first thirty-two entries of the Interrupt Descriptor Table.

Those entries handled processor exceptions.

This chapter extends the table by adding sixteen additional descriptors corresponding to the hardware interrupt vectors.

Conceptually, the table now appears as follows.

```text
Interrupt Descriptor Table

0  - 31   Processor Exceptions

32 - 47   Hardware IRQs

48 -255   Available
```

Notice that nothing fundamentally changes about the Interrupt Descriptor Table itself.

Only additional entries are added.

This demonstrates one of the strengths of the x86 interrupt architecture.

Processor exceptions and hardware interrupts are ultimately handled using the same mechanism.

---

# The Journey of a Hardware Interrupt

Once the Interrupt Descriptor Table has been extended, hardware devices can begin interrupting the processor.

The complete sequence is remarkably similar to the exception flow introduced in the previous chapter.

```text
Timer
   |
   v
PIC
   |
   v
CPU
   |
   v
IDT Entry
   |
   v
Assembly Wrapper
   |
   v
C Interrupt Handler
```

The principal difference lies only in the origin of the interrupt.

Instead of the processor detecting an internal fault, an external device initiates the sequence.

Everything after the interrupt reaches the processor is almost identical to exception handling.

---

# Acknowledging the Interrupt

One subtle but important detail distinguishes hardware interrupts from processor exceptions.

After servicing a hardware interrupt, the operating system must inform the Programmable Interrupt Controller that processing has completed.

This notification is known as the **End of Interrupt (EOI)**.

Without it, the PIC assumes the interrupt is still being serviced.

As a result, it refuses to deliver further interrupts from that source.

Conceptually, the exchange resembles the following.

```text
Device
   |
Interrupt
   |
   v
Operating System
   |
Process Request
   |
Send EOI
   |
   v
PIC Ready
```

Failure to send an End of Interrupt is one of the most common errors encountered when developing an interrupt subsystem.

The symptom is distinctive.

The first interrupt arrives successfully.

Every subsequent interrupt disappears.

---

# Giving the Kernel a Sense of Time

Among all hardware devices, none is more important during early operating system development than the **Programmable Interval Timer (PIT)**.

Unlike keyboards or disks, the timer generates interrupts continuously.

Its sole purpose is to notify the processor at regular intervals.

Conceptually, it behaves like a metronome.

```text
Tick
 |
Tick
 |
Tick
 |
Tick
 |
Tick
```

Every tick generates a hardware interrupt.

The operating system counts these interrupts to measure the passage of time.

Initially, the timer serves only to print an incrementing counter.

Later chapters will use exactly the same timer interrupts for process scheduling, sleeping threads, timeout management, and performance measurement.

The timer therefore provides the operating system with its first heartbeat.

---

# Programming the Timer

The Programmable Interval Timer operates from a fixed hardware clock.

The operating system cannot directly request an arbitrary frequency.

Instead, it programs a **divisor**.

The hardware divides its base frequency by this value to determine how often interrupts should be generated.

The relationship is straightforward.

```text
Base Clock
      |
      |
 Divide by N
      |
      v
Interrupt Frequency
```

A smaller divisor produces more frequent interrupts.

A larger divisor produces fewer interrupts.

Although the arithmetic is simple, selecting an appropriate frequency depends upon the operating system's goals.

Educational kernels commonly begin with frequencies between 50 and 100 timer interrupts per second.

---

# The First Periodic Interrupt

Once the timer has been configured, the processor begins receiving interrupts automatically.

Unlike the software interrupts used in the previous chapter, these occur without any explicit instruction from the kernel.

The sequence now resembles the following.

```text
Kernel Starts
      |
      v
Initialize Timer
      |
      v
Continue Executing
      |
      v
Timer Interrupt
      |
      v
Interrupt Handler
      |
      v
Resume Execution
```

The operating system has reached an important milestone.

Execution is no longer entirely sequential.

The processor periodically pauses normal execution to service hardware events before continuing where it left off.

This behavior forms the basis of every multitasking operating system.

---

# Hardware Interrupts in Long Mode

One of the surprising observations in this chapter is how little changes when moving from a 32-bit kernel to a 64-bit kernel.

The processor still receives interrupts through the Interrupt Descriptor Table.

The Programmable Interrupt Controller is programmed exactly as before.

The Programmable Interval Timer behaves identically.

Most of the differences arise from the execution environment rather than the hardware itself.

Long mode introduces larger interrupt frames, wider addresses, different calling conventions, and a new interrupt return instruction. Nevertheless, the overall architecture remains unchanged.

This illustrates an important design principle of x86 systems.

The hardware interface has remained remarkably stable across generations of processors. Once the processor has successfully entered long mode, interacting with legacy hardware differs very little from interacting with it in a 32-bit operating system.

---

# Looking Ahead

This chapter marks the point at which the kernel becomes event-driven. It no longer waits passively for instructions to complete. Instead, external hardware can interrupt execution at any moment, allowing the operating system to respond to the outside world.

The timer introduced here serves a purpose far beyond simply printing a counter. In later chapters, it will become the foundation for timekeeping, process scheduling, sleeping, preemption, and many other kernel services.

With interrupts now functioning correctly, the next major task is memory management. The identity-mapped page tables created during the bootstrap phase will be replaced with a complete virtual memory subsystem capable of allocating, protecting, and reclaiming memory dynamically.
